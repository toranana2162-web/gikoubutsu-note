# 歯科医院の「技工物管理ノート」を Claude Code でゼロから作って本番公開した話

> 種別：システム・ツール記事
> 公開URL：https://gikoubutsu-note.vercel.app

---

## TL;DR（要約）

- 歯科医院向けの **技工物（詰め物・被せ物などの外注品）管理アプリ** を、ビルド不要の静的SPA（HTML/CSS/バニラJS の3ファイル）で開発。
- データはブラウザの `localStorage` に保存し、**サーバー費用・ログイン不要・無料** で完結。
- 開発から Vercel への本番公開まで、**Claude Code** を相棒に進めた。
- 本番公開時のセキュリティヘッダ設定や、CLIの権限まわりのつまずきも Claude Code とその場で解決。

---

## 作ったもの

**「技工物管理ノート」** — 歯科医院で日々発生する「技工物（歯科技工所へ発注する被せ物・詰め物・入れ歯など）」の、**依頼〜納品状況をデジタル管理する** Webアプリです。

紙のノートで管理していた業務を、そのままの感覚でデジタル化することをコンセプトにしています。

### 主な機能

| 画面 | できること |
|------|-----------|
| **トップ** | 「今日の納品予定」「期限超過の未納品」「未納品合計」を一目で把握 |
| **一覧** | 患者名・納品予定日・状態での検索/絞り込み、納品済みチェック |
| **登録** | 印象日・患者名・製作物内容・対象歯・送るもの・納品予定日を入力 |

特徴的なのが **対象歯の入力**。Zsigmondy-Palmer（ジグモンディ・パーマー）方式の歯式チャートをタップで選択でき、「永久歯（1〜8）」「乳歯（A〜E）」を切り替えて混合歯列にも対応しています。`"UR6"`（右上6番）のようなコードで保持し、表示時に「右上6・7」のように人が読める形へ変換します。

納品ステータスは日付を基準に自動判定（`納品済み / 期限超過 / 本日納品 / 未納品`）し、期限超過は赤、本日納品は緑で視覚的に警告します。

## 誰のため

**歯科医院の受付・スタッフ** が対象です。

- 技工所への発注は数が多く、「いつ・誰の・何を頼んで・いつ納品予定か」を紙で管理すると抜け漏れが起きやすい。
- IT専任者がいない小規模クリニックでも使えるよう、**インストール不要・ログイン不要・無料** を絶対条件に設計。
- 技工所は1か所という現場の実態に合わせ、あえて技工所の登録・検索機能は省略してシンプルさを優先。

「現場スタッフが受け入れやすいこと」を最優先に、あえて **紙のノート風UI** にしています。

## 技術スタック

| 区分 | 採用技術 | 選定理由 |
|------|----------|----------|
| フロントエンド | **HTML / CSS / バニラJavaScript** | フレームワーク・依存パッケージ一切なし。学習/保守コストを最小化 |
| ビルド | **なし** | `index.html` を開けば動く。ビルド環境の保守が不要 |
| データ保存 | **localStorage** | サーバー費用ゼロ・ログイン不要。キー `gikoubutsu.records.v1` にレコード配列を保存 |
| ホスティング | **Vercel** | 静的サイトをゼロ設定で無料公開・HTTPS自動対応 |
| セキュリティ | **CSP / HSTS 等のHTTPヘッダ**（`vercel.json`） | 静的サイトでも堅牢に |

### 構成はたった3ファイル

```
index.html   … 画面構造（トップ / 一覧 / 登録 の3ビューをタブ切替）
styles.css   … 紙ノート風スタイル・レスポンシブ（640pxでPCレイアウト）
app.js       … IIFE。localStorage の CRUD・検索・状態判定
```

XSS対策として、ユーザー入力は必ず `escapeHtml()` を通して描画しています。

## 使った Claude Code の機能

今回の開発で実際に活躍した Claude Code の機能を挙げます。

### 1. `CLAUDE.md`（プロジェクト指示書）
リポジトリ直下に `CLAUDE.md` を置き、**アーキテクチャ・データモデル・制約方針** を明文化。これにより Claude Code が毎回プロジェクトの前提（「3ファイル構成を崩さない」「入力は `escapeHtml` を通す」など）を踏まえて動いてくれるので、方針のブレがなくなりました。

### 2. `@` 参照でのファイル指定
`@index.html` `@app.js` のように **ファイルを直接参照** させることで、該当箇所をピンポイントで読ませて修正・確認できました。

### 3. Plan Mode（設計→実装の分離）
いきなりコードを書かせず、**まず実装方針を立てさせてから着手** する進め方で、手戻りを削減。

### 4. 対話による意思決定
「複数端末で共有すべきか？」「本番公開はどのサービスを使うか？」といった **設計判断を選択肢付きで提示** してもらい、費用・手間・リスクを比較して決められました。特に「患者情報を扱うのでクラウド共有には認証が必須」という指摘は、独力だと見落としがちなポイントでした。

### 5. Git / デプロイのワークフロー
本番公開（Vercel）を Claude Code に任せ、**CLIのインストール〜プロジェクト作成〜デプロイ〜公開後の動作検証** まで一気通貫で実行。公開後は `curl` でHTTPステータス・セキュリティヘッダ・開発ファイルの除外まで自動検証してくれました。

### つまずき①：npm のグローバルインストールが権限で失敗

Vercel CLI を `npm install -g vercel` で入れようとしたところ、

```
npm error The operation was rejected by your operating system.
npm error It is likely you do not have the permissions ...
```

と `/usr/local` への書き込み権限で拒否されました。

**解決：** グローバルインストールをやめ、`npx vercel` で実行。ユーザー領域のキャッシュに展開されるため sudo 不要で動きました。以降のデプロイも `npx vercel --prod` で完結します。

### つまずき②：日本語のフォルダ名でプロジェクト名が作れない

プロジェクトフォルダ名が日本語（`技工物管理アプリ`）だったため、Vercel の自動プロジェクト名生成が通りませんでした（プロジェクト名は英数字とハイフンのみ）。

**解決：** `vercel link --project gikoubutsu-note` で **英字のプロジェクト名を明示的に指定** してリンク。

### つまずき③：セキュリティヘッダ（CSP）でアプリが壊れないか

患者名という個人情報を扱うため、公開時に厳格な **Content-Security-Policy** を設定したい。ただしCSPはインラインスクリプトやインラインstyleをブロックするため、既存コードを壊すリスクがあります。

**解決：** 適用前に `grep` で **インラインの `<script>`・`onclick`等のハンドラ・`style=` 属性が無いこと** を確認してから、`script-src 'self'` などの厳格なCSPを適用。公開後に `curl -I` でヘッダが効いていることも検証しました。

あわせて `.vercelignore` で `CLAUDE.md` などの開発用ファイルを公開対象から除外し、公開後に `CLAUDE.md` が `404` を返すことを確認しています。

### 設計上の判断：データ保存方式

「複数端末で共有したい」という要望を検討しましたが、

- `localStorage` は端末（ブラウザ）内保存のため端末間共有はできない
- クラウド共有（Supabase等）にすると患者情報の取り扱い上、**共有ログインによるアクセス制限が必須**

というトレードオフを比較。今回は **1端末完結・外部流出リスクの低さ・費用ゼロ** を優先し、`localStorage` のまま運用する判断としました（クラウド化はPhase2候補として保留）。

## まとめ

- 「フレームワークなし・ビルドなし・3ファイル」でも、現場の要件を満たす実用的な業務アプリは作れる。
- **Claude Code に `CLAUDE.md` でプロジェクトの制約を共有** しておくと、方針がブレず一貫した開発ができる。
- 実装だけでなく、**本番公開・セキュリティヘッダ設定・公開後の動作検証** までまとめて任せられるのが便利だった。
- 個人情報を扱うアプリでは「便利さ」と「安全性」のトレードオフを言語化して判断することが重要。今回はあえて機能を足さない選択も、根拠を持って選べた。
